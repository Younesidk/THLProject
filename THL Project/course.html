Lesson 1: State Class (Models/State.cs)

Code reference: Models/State.cs lines 3-23

Data Structure First

What problem this class solves

A finite-state automaton needs a canonical object for one node in the state graph. This class stores the state identity and whether it is the entry state or an accepting state.

How it fits in the bigger picture

Every transition points from one State to another, and the full automaton stores a collection of State objects. Recognition and determinization both depend on these flags when deciding start and acceptance behavior.

Key properties and methods and why they exist

  • Name: stable identifier used in maps and logs.
  • IsInitial: marks the automaton start state q0.
  • IsEnd: marks membership in accepting set F.
  • Constructor with flags: creates a fully defined state in one call.
  • Constructor with name only: supports incremental construction.
  • ToString override: improves readability in debugging and UI rendering.

CS concept behind it

In formal notation, this class represents an element of Q (the state set). IsInitial and IsEnd encode special subsets: one initial state and one or more final states. 

public class State
{
    public string Name { get; private set; }
    public bool IsInitial { get; private set; }
    public bool IsEnd { get; private set; }

    public State(string Name, bool isInitial, bool isEnd)
    {
        this.Name = Name;
        IsInitial = isInitial;
        IsEnd = isEnd;
    }
}

Lesson 2: Transition Class (Models/Transition.cs)

Code reference: Models/Transition.cs lines 3-20

Graph Edge Model

What problem this class solves

It models one labeled edge in the automaton graph: from state, read symbol, move to state. Without this class, transitions would be loose tuples and harder to validate or manipulate.

How it fits in the bigger picture

Automaton.AddTransition consumes Transition objects and stores them in the transition matrix. Both recognition and determinization depend on this normalized representation.

Key properties and methods and why they exist

  • From and To: capture source and destination states.
  • Symbol: transition label from alphabet Sigma.
  • Full constructor: convenient for complete edge insertion.
  • Partial constructor: allows workflows where destination is decided later.

CS concept behind it

This is a concrete representation of transition function delta. In NFA terms, multiple transitions with the same source and symbol are allowed, which is later represented by a list in each matrix cell. 

public class Transition
{
    public State? From { get; set; }
    public State? To { get; set; }
    public char Symbol { get; set; }

    public Transition(State from, State to, char symbol)
    {
        Symbol = symbol;
        From = from;
        To = to;
    }
}

Lesson 3: Automaton Class (Models/Automaton.cs)

Code reference: Models/Automaton.cs lines 5-127

Core Machine Representation

What problem this class solves

It stores the full automaton in memory: states, alphabet, and transitions. It also gives fast lookup for transition queries used by algorithms.

How it fits in the bigger picture

This is the shared data model for both algorithms in Logic. Recognizer reads from it. Determinizer reads an NFA Automaton and writes a DFA Automaton.

Key properties and methods and why they exist

  • transitionMatrix: 2D table indexed by state and symbol; each cell stores a list of reachable states.
  • StateIndexMap and SymbolIndexMap: convert names/symbols into matrix indices.
  • RebuiltMatrix: resizes matrix when states or symbols are added while preserving existing transitions.
  • AddState and AddSymbol: grow Q and Sigma safely.
  • AddTransition and GetTransition: write and read delta transitions.
  • IsDeterministic: verifies whether every cell has at most one destination state.
  • GetInitialState and GetFinalStates: expose q0 and F for algorithmic checks.

CS concept behind it

This class implements the formal automaton tuple (Q, Sigma, delta, q0, F). Because each matrix cell is a list, the same structure can represent both DFA and NFA behavior. 

public void AddTransition(Transition transition)
{
    if (!StateIndexMap.TryGetValue(transition.From.Name, out var rowNumber))
        return;

    if (!SymbolIndexMap.TryGetValue(transition.Symbol.ToString(), out var colNumber))
        return;

    if (transitionMatrix[rowNumber, colNumber] is null)
        transitionMatrix[rowNumber, colNumber] = new();

    transitionMatrix[rowNumber, colNumber].Add(transition.To);
}

public bool IsDeterministic()
{
    foreach (var state in StateIndexMap)
        foreach (var symbol in SymbolIndexMap)
            if (transitionMatrix[state.Value, symbol.Value]?.Count > 1)
                return false;

    return true;
}

Algorithm Output Envelope

What problem this class solves

A recognizer should return more than accepted versus rejected. This class bundles decision, reason, and execution trace so callers can inspect how and why the word was processed.

How it fits in the bigger picture

AutomatonRecognizer returns this object for every input word. It decouples algorithm internals from how another layer displays success or failure.

Key properties and methods and why they exist

  • Accepted: final language membership result.
  • RejectionReason: machine-readable and human-readable failure context.
  • StepLog: state-by-state trace for debugging and teaching.

CS concept behind it

This object makes recognizer output explainable. In automata tooling, trace data is critical for proving correctness and diagnosing malformed transition tables. 

public class RecognitionResult
{
    public bool Accepted { get; set; }
    public string? RejectionReason { get; set; }
    public List<string> StepLog { get; set; } = new();
}

Lesson 5: AutomatonDeterminizer Class (Logic/AutomatonDeterminizer.cs)

Code reference: Logic/AutomatonDeterminizer.cs lines 5-106

NFA to DFA Algorithm

What problem this class solves

It converts an NFA model to an equivalent DFA model so recognition can follow one deterministic transition per symbol instead of branching over multiple active states.

How it fits in the bigger picture

It is a preprocessing algorithm. If input automaton is already deterministic, it returns the same object. Otherwise, it builds a new DFA using subset construction and returns it to downstream logic.

Key properties and methods and why they exist

  • NFAtoDFA: main conversion routine with guard checks and full subset traversal.
  • stateMap dictionary: tracks already-created DFA states by canonical subset key.
  • queue: breadth-first processing of unexpanded NFA subsets.
  • GetKey: sorts state names so logically equal subsets map to the same key.

CS concept behind it

This is classical subset construction. Each DFA state is a set of NFA states. Acceptance propagates with existential rule: DFA state is final if any member NFA state is final. 

while (queue.Count > 0)
{
    var currentSet = queue.Dequeue();

    foreach (var symbol in alpha)
    {
        var nextSet = new List<State>();

        foreach (var nfaState in currentSet)
        {
            var reachable = NFA.GetTransition(nfaState, char.Parse(symbol));
            if (reachable is null) continue;

            foreach (var s in reachable)
            {
                if (s is null) continue;
                if (!nextSet.Any(x => x.Name == s.Name))
                    nextSet.Add(s);
            }
        }

        if (nextSet.Count == 0) continue;
        // Create missing DFA state and add transition...
    }
}

Lesson 6: AutomatonRecognizer Class (Logic/AutomatonRecognizer.cs)

Code reference: Logic/AutomatonRecognizer.cs lines 5-87

Word Recognition Engine

What problem this class solves

It executes the automaton on an input string and decides whether the string belongs to the language. It also captures a trace and structured rejection reasons.

How it fits in the bigger picture

This is the endpoint algorithm after model construction and optional determinization. It turns static structure into runtime behavior.

Key properties and methods and why they exist

  • Constructor injects one Automaton instance into recognizer logic.
  • RecognizeWord performs symbol-by-symbol simulation and logs each transition.
  • Early guard clauses reject null words, missing initial state, invalid symbols, and blocked transitions.
  • BuildRejectedResult centralizes error packaging for consistent outputs.

CS concept behind it

This is DFA-style simulation with linear scan over input length. Complexity is O(n) transition lookups. Current implementation picks first next state from a transition cell, so full NFA branching is not explored here. Determinization before recognition gives correct general behavior. 

for (int i = 0; i < word.Length; i++)
{
    char symbol = word[i];
    if (!automaton.IsValidSymbol(symbol))
        return BuildRejectedResult($"Invalid symbol: {symbol}", stepLog);

    var nextStates = automaton.GetTransition(currentState, symbol);
    if (nextStates is null || nextStates.Count == 0 || nextStates[0] is null)
        return BuildRejectedResult($"Blocked at state {currentState.Name} on symbol {symbol}", stepLog);

    var nextState = nextStates[0]!;
    stepLog.Add($"{currentState.Name} --{symbol}--> {nextState.Name}");
    currentState = nextState;
}

Lesson 7: Capstone - Full Execution Flow

Code references: Models/*.cs and Logic/*.cs only

From Input to Decision

What problem this capstone solves

It connects all classes into one coherent pipeline: input values become a model, algorithms run on that model, and the final acceptance decision is returned with trace information.

How it fits in the bigger picture

The capstone is the architecture-level view of the project core. It demonstrates separation of concerns: Models capture structure, Logic captures computation.

End-to-end flow: user input to result

  1. User input arrives as names, flags, alphabet symbols, transition definitions, and a test word.
  2. Model construction creates State objects and inserts them into an Automaton with symbols and transitions.
  3. Optional logic phase runs NFAtoDFA to normalize nondeterministic machines.
  4. Recognition phase runs RecognizeWord over the selected automaton.
  5. Result packaging returns Accepted, RejectionReason, and StepLog in RecognitionResult.

Key methods involved and why they exist

Model Build

State constructors, AddState, AddSymbol, AddTransition define Q, Sigma, and delta.

Normalization

AutomatonDeterminizer.NFAtoDFA creates deterministic equivalents for robust recognition.

Execution

AutomatonRecognizer.RecognizeWord performs symbol consumption and transition traversal.

Output

RecognitionResult carries final decision and explanatory trace for callers.

CS concept behind it

This is an implementation of language recognition pipeline. You can view it as a pure data model plus pure processing layer, a common architecture in compilers, parsers, and formal-language tools. 

// Flow sketch built only from available Models + Logic APIs
var automaton = new Automaton();
automaton.AddState(new State("q0", true, false));
automaton.AddState(new State("q1", false, true));
automaton.AddSymbol('a');
automaton.AddTransition(new Transition(new State("q0"), new State("q1"), 'a'));

var prepared = AutomatonDeterminizer.NFAtoDFA(automaton) ?? automaton;
var recognizer = new AutomatonRecognizer(prepared);

Lesson 8: AutomatonController Class (Logic/AutomatonController.cs)

Code reference: Logic/AutomatonController.cs

Facade & UI Bridge

What problem this class solves

The UI layer should never manipulate an Automaton, a Transition, or a RecognitionResult directly. AutomatonController sits between the UI and the Models/Logic layers and exposes only safe, validated operations. Every public method returns a simple value type so the UI has no reason to import automaton internals.

How it fits in the bigger picture

This class is the single entry point the UI calls for every user action: adding a symbol, adding a state, adding a transition, converting to DFA, recognizing a word, building the visual graph, and resetting the session. It owns the one live Automaton instance and replaces it when determinization runs.

Key properties and methods and why they exist

  • Private Automaton field: the controller is the sole owner of the automaton instance. No other class can corrupt it because no reference escapes the controller.
  • AddSymbol(char): validates that the symbol is not already in the alphabet before delegating to Automaton.AddSymbol. Returns a tuple so the UI knows whether to show an error.
  • AddState(string, bool, bool): checks for duplicate names and enforces the single-initial-state rule before creating a State object and calling Automaton.AddState.
  • AddTransition(string, char, string): resolves state names to State objects internally so the UI passes plain strings and never touches State references.
  • GetSymbolNames / GetStateNames / GetStates: read-only projections that let the UI populate combo boxes and grids without holding model references.
  • GetTargetStateNames(string, char): returns a list of strings for a given from-state and symbol, which is exactly what the transition matrix display needs.
  • IsDeterministic(): forwards the check so the UI can guard the convert button without knowing how determinism is tested.
  • ConvertToDFA(): calls AutomatonDeterminizer and replaces the internal automaton field with the result. The UI is not involved in this swap.
  • RecognizeWord(string): handles the empty-string edge case directly, then delegates to AutomatonRecognizer. Returns a plain tuple so the UI only needs to check a bool and display a string.
  • Reset(): replaces the internal automaton with a fresh instance, which is the only safe way to clear all state without leaving dangling references.
  • BuildGraph(): constructs and styles an MSAGL Graph object from the current automaton. All graph color constants live here so they never appear in UI code.

Design pattern behind it

This class is a Facade. A Facade wraps a complex subsystem (here: Automaton, Transition, AutomatonDeterminizer, AutomatonRecognizer, and MSAGL) behind a simpler interface. The UI only sees one class with plain method signatures. If any subsystem changes internally, only the controller needs to change, not the UI.

It also acts as a Service Layer in the sense that all validation and orchestration logic sits here, keeping both the model classes and the UI form clean.

Return tuple pattern

Every mutating method returns (bool Success, string Error) instead of throwing exceptions. This keeps error handling in the UI as a simple if-check rather than a try-catch block, which is appropriate for user-input validation that is expected to fail regularly. 

// AddSymbol: validate first, delegate second, report result
public (bool Success, string Error) AddSymbol(char symbol)
{
    if (automaton.GetSymbolNames().Contains(symbol.ToString()))
        return (false, $"'{symbol}' is already in the alphabet!");

    automaton.AddSymbol(symbol);
    return (true, string.Empty);
}
// AddState: two independent guard clauses before any mutation
public (bool Success, string Error) AddState(string name, bool isInitial, bool isFinal)
{
    if (automaton.GetStateNames().Contains(name))
        return (false, $"\"{name}\" is already in the Automaton");

    if (isInitial && automaton.InitExist())
        return (false, "There already exists an Initial State");

    automaton.AddState(new State(name, isInitial, isFinal));
    return (true, string.Empty);
}
// RecognizeWord: empty-string edge case handled before delegating
public (bool Accepted, string Reason) RecognizeWord(string word)
{
    if (string.IsNullOrEmpty(word))
    {
        var start = automaton.GetStates().FirstOrDefault(s => s.IsInitial);
        if (start != null && start.IsEnd)
            return (true, "Empty Word is Accepted");
        else
            return (false, "Empty Word is Rejected");
    }

    var recognizer = new AutomatonRecognizer(automaton);
    var result = recognizer.RecognizeWord(word);

    return result.Accepted
        ? (true, "Word is Accepted")
        : (false, result.RejectionReason);
}
// BuildGraph: all MSAGL construction isolated from the UI
public Graph BuildGraph()
{
    Graph graph = new Graph("Automaton Graph");

    var layoutSettings = new SugiyamaLayoutSettings();
    layoutSettings.EdgeRoutingSettings.EdgeRoutingMode =
        Microsoft.Msagl.Core.Routing.EdgeRoutingMode.Spline;
    graph.LayoutAlgorithmSettings = layoutSettings;

    foreach (var state in automaton.GetStates())
    {
        var node = graph.AddNode(state.Name);
        node.Attr.FillColor = GraphNodeFill;
        node.Attr.Color     = GraphNodeBorder;
        node.Label.FontColor = GraphNodeText;
        node.Attr.Shape     = Shape.Circle;

        if (state.IsEnd)
        {
            node.Attr.Shape = Shape.DoubleCircle;
            node.Attr.Color = GraphFinalState;
        }

        if (state.IsInitial)
            node.Attr.FillColor = GraphInitialState;
    }

    foreach (var fromState in automaton.GetStates())
        foreach (var symbolStr in automaton.GetSymbolNames())
        {
            char symbol = symbolStr[0];
            var targets = automaton.GetTransition(fromState, symbol);
            if (targets == null) continue;

            foreach (var toState in targets)
            {
                var edge = graph.AddEdge(fromState.Name, symbolStr, toState.Name);
                edge.Attr.Color      = GraphEdgeLine;
                edge.Label.FontColor = GraphEdgeText;
            }
        }

    return graph;
}

How the UI consumes the controller

User adds a symbol

UI calls AddSymbol(c). On false result it sets an error label. On true it adds a column to the grid and an item to the combo box.

User adds a state

UI calls AddState(name, isInit, isFinal). On true it adds a row header to the grid and entries to the From and To combo boxes.

User adds a transition

UI calls AddTransition(from, symbol, to). On true it finds the matching grid cell and writes the target state name into it.

User converts to DFA

UI calls IsDeterministic first. If false it calls ConvertToDFA then RefreshMatrix and RefreshGraph to rebuild both visual components from the new automaton state.

User tests a word

UI calls RecognizeWord on every text change. It sets the label text and color based on the returned bool and reason string.

User resets

UI calls Reset then clears all input controls. The controller discards the old automaton and the UI rebuilds from scratch.
